## 一、赛道与定位

### Q65：你们报的是哪个赛道？为什么？

**答**：我们报的是 **Genesis 创世纪赛道（生 · Let there be Life）**。

理由很直接——ClawTree 不是一个纯 AI 项目（不是只做 LLM 应用），也不是一个纯 Web3 项目（不是只做智能合约）。我们的核心价值恰恰在于 **AI 和 Web3 的交叉融合**：

- AI Agent 负责发现机会、匹配合作、生成方案
- Web3 链上存证负责让合作成果可验证、不可篡改
- 人审负责品牌和合规的最终把关

这恰好是 Genesis 赛道鼓励的"前所未见的跨界玩法"。而且赛事明确说"AI × Web3 融合项目获得额外加分"——我们不需要强行融合，因为融合本身就是我们的产品逻辑。


### Q66：如果评委说"你们也可以报 AI 赛道或 Web3 赛道"，你怎么回应？

**答**：技术上确实可以，但拆开来看：

- **如果只报 AI 赛道**：六个 Agent Pipeline 是亮点，但链上存证的价值就被忽略了。ClawTree 不只是"用 AI 生成邮件"，Proof of Work 是可验证合作的核心差异化。
- **如果只报 Web3 赛道**：三个 Solidity 合约 + TronLink 集成是亮点，但 AI 驱动的采集、匹配、外联全链路才是解决真实问题的部分。没有 AI，合约只是存哈希的空壳。

Genesis 赛道不是为了"规避竞争"选的，而是因为 **AI × Web3 融合不是我们的 feature，是我们的 DNA**。


### Q67：HTX DAO 是主办方之一，你们和 HTX DAO 有什么关系？

**答**：HTX DAO 是 Genesis 黑客松的联合主办方，也是我们项目生态集成的重要方向。README 中明确列出了四个已实现的集成点：

1. **$HTX 行情数据**：运营 Dashboard 调用 HTX 公开 API（`api.huobi.pro/market/detail/merged?symbol=htxusdt`），展示 $HTX/USDT 实时价格卡片——不是静态截图，是真实 API 调用。
2. **Genesis 活动情报**：AI 助手的审核知识库和 golden dataset 收录了 HTX Genesis Hackathon 作为带来源引用的生态活动。
3. **TRON 证明层**：三个 Solidity 合约部署在 TRON Nile 测试网，TronLink 钱包交互，链上凭证可跳转 Nile 区块浏览器验证。
4. **生态增长工作流**：Genesis、Grant、黑客松和开发者计划信号 → 校园 Campaign → 合作方案 → 审批触达 → 影响力报告。

扩展路径还包括：在 B.AI 算力上运行 Agent 工作负载、将 HTX 生态项目转化为可复用的 Campaign 工作区、隐私安全的赞助方 Grant 里程碑证明、完成法律审核后使用 $HTX 承载已审批的活动预算或生态奖励。

我们不是在"蹭"HTX DAO 的品牌，而是在 HTX 生态的 TRON 链上构建真实可用的产品功能——关键生态贡献不是一个价格组件，而是一层把生态项目转化为可衡量增长的运营系统。


## 二、主办方资源利用

### Q68：你们怎么用 b.ai 的算力资源？

**答**：赛事提供了价值 10 万美元的 b.ai 算力支持。我们的使用规划：

1. **Agent Pipeline 推理**：六步 Agent 流程（classify → dedup → compliance → match → proposal → reply）的 LLM 推理全部迁移到 b.ai 算力上运行，替代当前的 DeepSeek API 调用。
2. **批量邮件生成**：当一个 Campaign 需要对接 50+ 所高校时，并行生成个性化邮件对算力需求较大，b.ai 的 GPU 集群可以显著加速。
3. **RAG 向量检索**：AI 客服的知识库检索和 Embedding 计算在 b.ai 上运行，降低延迟。
4. **模型微调**：未来如果要针对高校合作场景微调一个专属模型，b.ai 的算力是必要条件。

一句话：b.ai 让 ClawTree 从"调用外部 API"升级为"拥有专属算力的 AI 操作系统"。


### Q69：OpenCSG、TintinLand、OpenCity 这三个主办方和你们有什么关系？

**答**：

- **OpenCSG（开源中国/社区）**：我们项目本身就是开源的（GitHub 公开仓库），与 OpenCSG 的开源理念完全一致。未来社区运营和开发者关系可以依托 OpenCSG 的社区资源。

简单说：这三位主办方的资源是 ClawTree 潜在的合作场景和分销渠道，不是"跟我没关系的主办方 Logo"。

## 三、赛事评审标准对齐

### Q71：评审标准里有"AI 技术创新性"，你们的技术创新在哪？

**答**：不是"用了 AI"就叫技术创新。ClawTree 的 AI 创新在于 **可验证的证据驱动 Agent 协议**：

1. **每个 AI 输出都有证据锚点**：不是让模型自由发挥然后相信它的"置信度"，而是要求每个声明（claim）必须带上源引用（sourceIds）。
2. **严格 JSON Schema 校验**：六个 Agent 任务都有完整的 JSON Schema，无效输出直接回退到确定性逻辑——不是"AI 不行就报错"，而是"AI 不行也有兜底"。
3. **安全层内嵌**：提示词注入检测不是在"上线后修补"，而是写在 Agent 协议层的 8 类正则规则——所有外部内容先过安全层再进模型。
4. **优雅降级**：LLM 挂了、预算超了、API key 没配——任何一种情况都不会导致系统不可用，自动切换确定性回退。

这跟"调个 API 用 ChatGPT 写邮件"有本质区别。


### Q72：评审标准里有"Web3 原生特性"，你们怎么体现？

**答**：三个层面：

1. **去中心化验证**：链上 Proof 不依赖 ClawTree 的服务器——任何人拿着哈希可以去 TRON Nile 区块浏览器独立验证。这是"Don't trust, verify"的 Web3 精神。
2. **自主托管**：TronLink 钱包由运营方自己管理私钥，ClawTree 不托管任何人的钱包或资产。
3. **不可篡改的审计轨迹**：已上链的合作记录不可删除、不可修改——给赞助方和生态伙伴的不是"我们的报告"，而是"链上的事实"。

同时我们明确不做的事：
- 不发币、不搞 tokenomics 噱头
- 不把用户数据上链（隐私保护优先于去中心化）
- 不为 Web3 而 Web3——只有在产生真实产品价值的地方才引入链上元素

README 中列出了九条安全原则，每一条都在代码中可验证：
1. 每个公开事实保留来源、发布时间和验证上下文
2. AI 建议和已验证事实以不同状态展示
3. 发布、发送、承诺和 live proof 都需要人工操作
4. 触达遵循一个组织、一份草稿的原则；BCC 不是个性化
5. 只使用有公开证据支持的机构联系点
6. 公开 API 不暴露运营数据或个人数据
7. 证明载荷绝不包含联系数据、消息内容、回复或提示词
8. 金融和体育内容仅用于教育：不提供博彩、价格预测、投资建议、收益承诺或结果保证
9. 外部服务必须具备基于 fixture 的失败模式


### Q72.5：Proof 白名单具体包含哪些字段？怎么保证隐私？

**答**：Proof 的规范化流程在 README 中有明确定义。

**上链的字段**（全部为公共摘要信息）：
- `payloadVersion`、`workspaceId`、`draftId`
- `universityName`、`eventTitle`
- `approvedBy`、`approvedAt`、`approvalStatus`

**明确排除的字段**（绝不上链）：
- 邮箱地址、联系人姓名
- 邮件正文、回复原文
- 内部备注、Prompt
- 凭证和私钥

技术实现：先对上述白名单字段做 JSON 规范化（排序 key、去除空格），再做 SHA-256 哈希，最后将哈希锚定到链上。重复的规范化输入产生相同的哈希——这是确定性证明，可以被任何人独立验证。链上不存原文、不存个人信息、不存可逆数据。


### Q73："Vibe Coding 实践者"加分项，你们符合吗？

**答**：部分符合。

我们确实在开发过程中使用了 AI 编程工具辅助——样板代码生成（Django Model/Serializer）、测试用例扩写、文档翻译。但核心逻辑——Agent 安全层、隐私白名单、状态机规则、确定性回退——是人工设计的。

这恰好反映了 ClawTree 的产品哲学：AI 做规模化（重复劳动），人做判断和设计（安全边界）。我们用 Vibe Coding 加速开发，但绝不让 AI 决定安全策略。

4. **开源影响力**：ClawTree 开源，其他开发者可以 fork 代码在 b.ai 上跑——b.ai 的算力通过 ClawTree 触达更多开发者。

## 五、Demo Day 实战问题

### Q78：现场 Demo 只有几分钟，如果网络卡了/TronLink 崩了怎么办？

**答**：我们有五层保障，与 README 中描述的优雅降级策略一致：

1. **在线部署（leaf666.xyz）**：走真实 HTTP 请求，展示完整流程。
2. **离线 Demo（npm run demo）**：读本地 fixture，零外部依赖，网络断了也能跑完全流程。README 明确：默认演示是确定性的，不需要模型密钥、数据库、邮箱、钱包或公网环境。
3. **确定性回退**：LLM 不可用时自动切模板匹配，钱包不可用时用 mock proof，数据库不可用时用 fixture 数据——每一步都有 fallback，且 fallback 行为在 README 中有完整的矩阵表格。
4. **Smoke Test 已验证**：`npm run smoke` 覆盖了 HTTP golden path（health → draft → approval → proof），全部通过。
5. **视频备份**：提前录好 3 分钟端到端 Demo 视频，应急情况下直接播放。

在黑客松 Demo 中，"能跑"比"跑得快"重要一百倍。我们的架构设计从一开始就考虑了 Demo 的可靠性。


### Q78.5：Demo 为什么不直接用真实 SMTP 发送邮件？

**答**：SPEECH2.md 中明确解释了这一点——本地 Demo 默认使用 Django Console Email Backend。

这样设计的原因：
1. **避免误发送**：现场点击"批准"不会误发给真实高校，而是打印在后端终端——评委可以看到邮件确实生成了、审批流程确实走通了，但不会产生不可控的副作用。
2. **证明路径完整**：虽然用的 Console Backend，但代码中的 SMTP 发送路径已经接好——生产环境只需改一个环境变量就能切换为真实发送。
3. **保留审批门槛**：即使在 Console 模式下，人工审批流程依然强制执行——这证明了"AI 不能自行发送"的安全边界是代码级别的约束，不是口头承诺。

一句话：Demo 展示的是完整的技术链路，但故意不接真实发送——这是对高校的尊重和对自身产品哲学的实践。


### Q79：Demo 中最想让评委记住的一个时刻是什么？

**答**：**外联审批 → 链上存证的那个瞬间**。

因为那一刻浓缩了 ClawTree 全部的产品哲学——README 中定义的"两个相互增强的产品闭环"同时在这个瞬间体现：

> **可信内容中继**：公开来源 → 收集、分类、去重、风险审查、人工审核 → 发布为可理解的校园内容
> **从机会到方案**：已验证活动 → 可解释的匹配 → 三档合作方案 → 单收件人草稿 → 审批记录 → 活动证明

三秒内展示了：AI（生成）→ 人审（批准）→ Web3（存证）→ 隐私保护（只哈希公开字段）。

如果评委只记住一个画面，我们选这个。


### Q79.5：README 里提到"3 分钟演示"，具体是怎么设计的？

**答**：README 中列出了 7 步确定性演示路径，不需要任何外部依赖：

1. 老师打开 `/user`，无需浏览 X，也能看到带来源的校园摘要和 AI/Web3 机会
2. AI 助手基于审核知识库回答问题，展示引用，不确定或高风险问题转人工
3. 运营人员切换到 `/admin`，查看采集、去重、风险标签、编辑修改和审批状态
4. ClawTree 将校园机会与已审核的组织能力进行匹配
5. Proposal Agent 生成轻量、中度、深度三档合作方案，带声明级证据
6. Outreach Copilot 为单个校园创建独立草稿——明确审批前不发送
7. ClawTree 将公开活动摘要哈希到模拟 TRON Nile 证明，排除联系人和邮件正文

三个产品界面各司其职：
| 界面 | 用途 |
|------|------|
| `/demo` | 确定性端到端黑客松故事演示 |
| `/user` | 面向老师和学生的公开门户 |
| `/admin` | 运营、编辑审核、方案和触达管理 |

这不是七个孤立的功能点，而是一条从"老师看到活动"到"运营审批触达"到"链上存证"的完整旅程。


### Q80：Demo 路径为什么选"大树财经高校行"而不是别的？

**答**：因为它是"最窄但最真实"的验证场景：

- 大树财经确实在推广全球高校行计划
- 高校活动确实分散在各校官网和社群中
- 运营团队确实需要 AI 辅助做规模化外联

一个真实的痛点场景比一个华丽的虚构场景更有说服力。"大树财经"是参考案例，README 明确声明："大树财经并不被描述为 ClawTree 的客户、所有方或背书方。底层工作区、Agent、数据和证明架构，面向任何媒体品牌、生态团队、高校网络或活动运营方设计。"换个工作区，同一套系统可以服务任何生态项目。


### Q80.5：project.md 里提到"世界杯 × 广州高校行"主题包，这是什么？

## 七、刁钻追问（赛事特供）

### Q83：你们项目叫 ClawTree（树爪智动），和大树财经（TreeFinance）就差一个"大"字，你们到底是什么关系？

**答**：没有股权、投资或从属关系。

大树财经是 ClawTree 的**参考演示场景**——就像 Notion 的 Demo 里会展示一个"产品团队的工作空间"一样。选择大树财经是因为：

1. 他们真实在推进高校行计划，有真实的运营痛点。
2. 他们有公开的 X 账号和推文数据，可以用来验证采集和 AI 分析链路。
3. 和大树财经无关——ClawTree 的 workspace 架构天然支持任何品牌，大树财经是第一个但不是唯一一个。

名字的巧合只是巧合。ClawTree = Claw（爪）+ Tree（树），寓意"用 AI 的爪子抓住碎片化信息，生长成合作网络"。


### Q84：如果赛事结束后没人继续维护这个项目怎么办？

**答**：我们设计了这个项目让它"没人维护也能跑"：

1. **离线 Demo 是永久的**：`npm run demo` 不依赖任何外部服务，只要 Node.js 还在就能跑。五年后的开发者依然可以 clone 下来看到完整的项目演示。
2. **确定性回退不依赖 LLM**：即使 DeepSeek 关门了，确定性模板依然能输出合理的结果。
3. **GitHub 上的代码和文档是完整的学习资源**：即使团队不再维护，其他开发者可以 fork 和继续。

但我们不打算"丢下不管"。ClawTree 解决的是真实问题，我们会在黑客松后继续迭代。


### Q85：（终极刁钻）如果我是投资人，你现在敢让我投钱吗？敢的话估值多少？不敢的话为什么？

**答**：敢让你投一笔小钱（天使轮级别），但不敢让你投一笔大钱（A 轮级别）。

- **敢要小钱**：因为产品方向验证了（真实痛点）、技术验证了（测试通过）、团队验证了（黑客松产出可见）。一笔天使资金足以完成广州 3-5 所高校的真实试点。
- **不敢要大钱**：因为商业模式还没有数据支撑（workspace 订阅定价没验证）、市场规模还没量化（多少生态项目愿意付费未知）、团队还没经历过"从工具到公司"的转变。

估值不是这个阶段该讨论的话题——我们更希望你作为"第一个付费客户"而不是"投资人"的身份参与进来。等我们有了 10 个付费 workspace 之后，再谈估值会更有底气。
