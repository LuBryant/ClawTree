## 一、黑客松执行与团队协作

### Q41：这三天黑客松你们具体做了什么？哪些是之前就有的？

**答**：我们在黑客松期间聚焦三件事：

1. **打通 Django 后端全链路**：活动采集 API → AI 外联邮件生成 → 外联审批状态机 → Pipeline 四步自动化调度 → 链上存证锚定。每个环节都接入了真实数据库。
2. **完善 Agent 安全层**：提示词注入检测（8 类正则）、引用覆盖验证（claim 级 sourceId 校验）、隐私白名单序列化器。12 个对抗性测试全部通过。
3. **HTX 生态集成**：$HTX 行情卡片、TRON Nile 合约部署、TronLink 钱包交互、Genesis 生态知识入库。

之前的代码基线：Next.js 前端框架、Agent Schema 定义、离线 Demo fixtures。黑客松期间我们把"看起来能跑"变成了"真实数据流可验证"。


### Q42：你们团队几个人？怎么分工的？

**答**：（根据实际情况填写。参考框架如下）

- **全栈 × 1**：Django 后端模型、API、Pipeline、序列化器；Next.js 路由和 Agent Runtime
- **合约 × 1**：Solidity 三个合约开发、Hardhat 测试、TRON Nile 部署、TronLink 集成
- **AI/Prompt × 1**：6 个 Agent Schema 设计、安全层、RAG 知识库、确定性回退
- **产品/演示 × 1**：Demo 脚本、PPT、演讲稿、评委 QA 准备


### Q43：代码里哪些是 AI 辅助写的？比例大概多少？

**答**：AI 辅助了约 40% 的代码——主要是样板代码（Django Model/Serializer 定义）、测试用例生成和文档。核心逻辑全部由人工设计：Agent 安全层、隐私白名单、状态机转换规则、确定性回退策略、链上 Proof 的 canonical hash 逻辑。这些不是 AI 能"理解"并独立设计的。

我们项目的 irony 是：我们本身就是一个让 AI 辅助但不替代人的系统——所以我们在开发过程中也在践行同样的哲学。


## 二、数据与隐私深度追问

### Q44：你们采集高校活动数据，有没有法律风险？

**答**：我们的数据采集遵循几个原则：

1. **只采集公开数据**：活动标题、时间、地点、公开联系方式——这些都是高校主动公开发布在官网上的信息。
2. **保留来源和采集时间**：每条数据都有 source_url、采集时间戳和内容哈希，可追溯、可审计。
3. **不采集个人数据**：我们不采集学生个人信息、不爬取需要登录才能访问的页面、不采集受 robots.txt 禁止的内容。
4. **提供 opt-out 机制**：高校可以通过联系我们要求移除特定活动的展示。

在法律定位上，我们更接近"搜索引擎索引公开网页"而非"爬取私密数据"。


### Q45：如果高校要求删除数据怎么办？

**答**：三步处理：

1. **公开端立即下架**：将对应的 EditorialReview 状态设为 `unpublished`，公开 API 不再返回该条数据。
2. **管理端保留审计轨迹**：原始采集数据不会物理删除（ContentItem 不可变设计），但标记为 `opt_out` 状态，不再参与任何 AI 分析或外联流程。
3. **链上记录不受影响**：已上链的只是公开摘要哈希，不包含可定位到具体高校的隐私信息，且区块链本身不可篡改——这是设计之初就考虑到的。


### Q46：你们怎么处理用户（老师、学生）的隐私？

**答**：

1. **公开端不需要注册**：浏览活动不需要登录，我们不收集浏览者的个人信息。
2. **AI 客服不存储对话**：客服对话仅用于当次问答，不持久化到数据库。
3. **合作意向转人工时才收集联系方式**：用户在 `/user/cooperate` 页面主动提交合作意向时，才会要求提供联系方式——且明确告知用途并取得同意。
4. **公开 API 零 PII**：公开端序列化器白名单不包含任何联系方式字段。


## 三、扩展性与工程深度

### Q47：系统能撑多少数据量？SQLite 能用于生产吗？

**答**：

本地 Demo 用 SQLite 是为了零配置启动。生产环境有明确的升级路径：

```python
# Django settings.py 支持环境变量切换
DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.mysql',  # 生产环境
        'NAME': os.environ.get('DB_NAME'),
        'USER': os.environ.get('DB_USER'),
        'PASSWORD': os.environ.get('DB_PASSWORD'),
        'HOST': os.environ.get('DB_HOST'),
        'PORT': os.environ.get('DB_PORT', '3306'),
    }
}
```

数据库层面：
- UniversityEvent 表在 source_url 上有唯一索引（去重）
- ContentItem 表在 content_hash 上有唯一索引（去重）
- 采集频率受 daily_budget_cents 限制
- IngestionRun 记录每次采集的游标和分页

预估：SQLite 足以支撑 10 万条活动 + 100 万条内容。超过这个量级切换 MySQL 即可，不需要改任何业务代码。


### Q48：如果同时有 100 个高校活动需要发送外联邮件，系统怎么处理？

**答**：

1. **Pipeline Step 3（邮件生成）**：逐条生成，每封之间有 0.3 秒间隔（`time.sleep(0.3)`），避免打爆 LLM API。
2. **Pipeline Step 4（审批发送）**：逐封审批和发送，有 `max_count` 上限控制。
3. **idempotency_key**：每条外联记录有幂等键，重复请求不会生成重复草稿。
4. **失败隔离**：单封邮件生成或发送失败，不影响其他邮件的处理（`try/except` 包裹每一封）。
5. **限速设计**：生产环境中接 SMTP 后会加上 `sendgrid` 或 `ses` 的限速策略，避免被邮件服务商封禁。


### Q49：多工作区怎么隔离？一个工作区能看到另一个的数据吗？

**答**：严格隔离，三层防护：

1. **数据库层**：所有核心模型（UniversityEvent、OutreachDraft、ContentItem 等）都有 `workspace` 外键，查询时强制按 workspace 过滤。
2. **API 层**：所有 ViewSet 继承自 `WorkspaceScopedQuerysetMixin`，自动根据请求中的 workspace slug 过滤数据。
3. **公开端**：不同工作区的公开端是不同的 URL 路径（如 `/user/treefinance` vs `/user/other-brand`），公开端只展示当前工作区的内容。

一个工作区的管理员无法通过任何 API 访问另一个工作区的运营数据。


## 四、AI 深度与模型策略

### Q50：为什么用 DeepSeek 而不是 GPT-4 或 Claude？

**答**：

1. **性价比**：DeepSeek 的 API 价格约为 GPT-4 的 1/30，对于需要批量处理活动分类和邮件生成的生产场景，成本是首要考虑。
2. **中文能力**：DeepSeek 对中文高校活动描述的理解和中文邮件生成能力在测试中表现优异。
3. **OpenAI 兼容协议**：DeepSeek 使用与 OpenAI 完全兼容的 API 格式，切换成本为零——改一个环境变量即可换模型。
4. **不绑定单一供应商**：我们的架构支持任何 OpenAI-compatible provider（Qwen、Claude API、GPT-4 等），运维团队可以根据成本和效果灵活切换。


### Q51：你们考虑过用本地模型（如 Ollama）吗？

**答**：考虑过，但目前不适合：

- **优势**：零 API 成本、数据不出本地、不受外部 API 限流影响
- **劣势**：需要 GPU 服务器、中文能力弱于 DeepSeek、运维复杂度高

我们的折中方案是"确定性回退"——在 LLM 不可用时，基于规则模板完成核心任务。这比本地模型的体验更可控，也比 API 调用更省钱。未来如果本地模型的中文能力追上来了，我们在 agent-provider 层加一个 Ollama adapter 即可，架构不需要改动。


### Q52：你们的 Prompt 是怎么设计的？能看看吗？

**答**：Prompt 设计遵循几个原则：

1. **结构化输出约束**：对于 JSON 模式任务，Prompt 明确要求"只返回 JSON，不返回其他内容"，并给出完整的 JSON Schema。
2. **安全护栏内嵌**：在 System Prompt 中明确禁止以下行为：
   - 不要编造未确认的合作细节
   - 不要承诺具体金额、曝光量或投资回报
   - 不要建议跳过人工审批流程
3. **角色边界清晰**："你是一个高校合作外联助手"——让模型清楚自己的定位和边界。
4. **Fallback 提示**：Prompt 中提供的信息足以让确定性模板也能工作。

具体的 Prompt 模板在代码中是常量（`EMAIL_SYSTEM_PROMPT`、`EMAIL_PROMPT_TEMPLATE` 等），欢迎在 GitHub 仓库中查看。


## 五、商业与运营深度

### Q53：你们怎么获取第一批用户（高校）？

**答**：不做"冷启动推广"，而是走"内部运营工具→生态平台"的渐进路线：

1. **第一阶段（当前）**：大树财经自身运营团队使用管理端，替代人工刷信息、写邮件的工作流。这是最直接的内部验证。
2. **第二阶段**：通过大树财经已有的高校合作关系，邀请 3-5 所广州高校的老师试用公开端——他们不需要"注册"，只需要打开网页看活动信息。
3. **第三阶段**：当公开端有足够多的高校活动后，其他高校会自然发现"别的学校在用什么工具"——跨校网络效应启动。
4. **第四阶段**：黑客松、协议方、Grant 项目等生态伙伴创建自己的工作区。


### Q54：运营一个工作区需要几个人？

**答**：

| 角色 | 人数 | 职责 |
|------|------|------|
| 内容运营 | 1-2 人 | 审核 AI 分类结果、编辑摘要、管理发布 |
| 外联运营 | 1 人 | 审核 AI 生成的邮件、批准/驳回/修改 |
| 技术运维 | 0.5 人 | 监控采集 Pipeline、处理失败重试 |

相比纯人工方式（3-5 人全职刷信息+写邮件），ClawTree 可以将运营效率提升 3-5 倍。核心不是"裁员"而是"让同样的人覆盖更多高校"。


### Q55：你们考虑过做移动端吗？

**答**：当前优先 Web 端的原因：

1. **运营端**（管理员）的复杂操作——审批邮件、编辑内容、查看 Pipeline 日志——在手机上体验不好，Web 端是最佳选择。
2. **公开端**（老师/学生）的浏览场景适合移动端，但我们用 Next.js 的响应式设计先覆盖移动浏览器，不需要单独开发 App。
3. **未来方向**：如果有强需求，可以考虑微信小程序——因为中国高校老师和学生重度使用微信。但那是产品验证之后的事。


## 六、风控与异常处理

### Q56：如果 AI 生成的邮件包含不实信息（幻觉），人审没看出来发出去了怎么办？

**答**：

1. **降低幻觉概率**：我们的 AI Agent 必须引用 sourceIds，邮件生成基于真实活动数据（标题、日期、描述），不能凭空编造。
2. **Prompt 约束**：System Prompt 明确要求"基于提供的活动信息撰写，不要添加未确认的细节"。
3. **人审是最后防线**：审核界面展示活动原始信息（标题、日期、描述）和 AI 生成邮件并排对比，降低审核遗漏的概率。
4. **回撤能力**：如果发现已发送邮件有问题，可以通过同一外联渠道发送修正邮件，并在 OutreachDraft 中记录。
5. **链上记录不受影响**：Proof 只锚定公共摘要哈希，不涉及邮件正文内容。

坦率说，没有任何系统能 100% 防止错误——但我们的多层防护设计把概率降到了商业可接受的水平。


### Q57：如果有人恶意提交大量外联请求（刷接口）怎么办？

**答**：

1. **idempotency_key**：每个外联请求都有幂等键，重复请求不会创建重复草稿。
2. **频率限制**：Pipeline 有 `max_count` 和 `schedule_time` 限制，不会无限生成。
3. **预算控制**：每日 LLM API 调用有 `daily_budget_cents` 预算上限，超出后自动回退到确定性模板。
4. **审批门槛**：即使绕过了生成限制，邮件也必须逐封经过人工审批才能发送。

攻击者能做的最多就是生成一堆草稿（不花钱、不发邮件），运营人员一眼就能看出来并批量驳回。


### Q58：采集 Pipeline 跑失败了怎么发现？怎么恢复？

**答**：

1. **IngestionRun 记录**：每次采集执行都有完整的运行记录——状态（running/succeeded/failed）、开始时间、结束时间、采集数、新增数、重复数、失败数和耗时。
2. **游标恢复**：SourceConnector 保存游标（cursor），失败的采集任务可以从上次中断的位置继续，不会重复处理。
3. **告警机制**：AgentAlert 模型支持基于阈值的告警（连续失败 N 次、失败率超过 X%、成本异常），触发后通知运营人员。
4. **手动触发**：Pipeline 支持手动触发，运营人员可以随时重新执行某个步骤。


## 七、生态与长期价值

### Q59：你们这个项目五年后会长成什么？

**答**：

我们看到的终局不是"一个更好的 CRM"，而是**跨组织合作的操作系统层**：

- **短期（1 年）**：内部运营工具 → 多个媒体和生态团队使用
- **中期（2-3 年）**：双向网络——高校主动来平台发布活动，赞助方在平台上寻找合作机会
- **长期（5 年）**：任何组织可以通过 ClawTree 发现合作信号、匹配合作伙伴、执行外联、证明成果。AI 不是聊天机器人，而是贯穿整个合作生命周期的运营基础设施。


### Q60：你们开源了，怎么防止别人抄？

**答**：

1. **开源是战略选择**：生态项目（黑客松、协议方、高校）天然偏好开源基础设施，因为需要审计、定制和信任。闭源反而限制了 adoption。
2. **壁垒不在代码，在数据和网络**：代码可以复制，但：
   - 持续采集的活动数据飞轮不能复制
   - 高校使用的网络效应不能复制
   - HTX/TRON 生态的集成深度不能复制
3. **开源促进行业标准**：如果 ClawTree 的 Agent Schema 和 Proof 格式成为生态标准，那反而是最强的护城河。
4. **商业价值在服务层**：我们开源的是引擎，收费的是托管服务、工作区订阅和赞助方报告。


### Q61：你们要求评委记住的一件事是什么？

**答**：ClawTree 不是一个"用 AI 写邮件"的工具——它是一个把碎片化的公开信号变成可验证的合作成果的**操作系统**。AI 负责规模化，人负责品牌承诺，链上负责可信执行。三个加起来，才是我们认为 AI × Web3 应该有的样子：不是替代人，而是让人的判断力覆盖更大的范围。


## 八、反问与自省（加分项）

### Q62：你们自己觉得项目最大的弱点是什么？

**答**：诚实地说，三个弱点：

1. **冷启动依赖**：目前围绕大树财经一个 Demo 场景，多工作区还需要更多真实数据验证。
2. **AI 依赖外部 API**：DeepSeek 的可用性和价格不在我们控制范围内。我们有确定性回退，但回退效果不如 LLM。
3. **链上存证的市场教育**：很多合作方（尤其是传统高校）不理解"链上哈希"的价值，需要教育市场。

但这三个弱点也是我们的 roadmap——多工作区、模型无关架构、用户教育——每个都在计划之中。


### Q63：如果重新来一次黑客松，你会做什么不同的决定？

**答**：

1. **更早打通端到端**：我们花了不少时间在 Agent Schema 设计和安全层上，如果重来，会先用最简单的 Pipeline 跑通一条完整的"采集→生成→审批→证明"链路，再逐层加固。
2. **更早录 Demo 视频**：现场演示总有不确定性，提前录好一个 3 分钟的端到端 Demo 视频作为备份会更稳妥。
3. **多准备一个备用场景**：大树财经是很好的参考案例，但如果评委问"其他场景呢"，能现场切换一个黑客松/Grant 项目的 Demo 会更有说服力。


### Q64：（评委反问）你觉得我们为什么要选你们？

**答**：三个理由：

1. **不是 PPT 项目**：56 个测试、6 个合约测试通过，代码可运行、可验证。这在黑客松中不是特别多但也不是每个项目都有。
2. **解决真实问题**：高校合作外联是每个 Hackathon、协议方、Grant 项目都面临的实际痛点——不是我们想象的场景。
3. **AI × Web3 的融合不是噱头**：AI 做规模，Web3 做信任——两者在 ClawTree 中的结合有产品逻辑支撑，不是硬凑技术栈。

如果只选一个 AI 项目和只选一个 Web3 项目都很容易。但如果要选一个真正把 AI 和 Web3 融合出产品价值的项目，我们认为 ClawTree 是一个值得考虑的选择。
